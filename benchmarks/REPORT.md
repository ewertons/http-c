# http-c benchmark report

Generated by `benchmarks/run_bench.sh` inside the
`http-c-build:ubuntu24` Docker image.

## Test setup

- Workload: `GET /` returning `hello\n` (6 bytes) over HTTPS
- Both servers respond with explicit `Content-Length` so wrk can use
  HTTP/1.1 keep-alive on persistent connections.
- TLS: OpenSSL 3.0.13 (server cert is the same self-signed RSA-2048 cert
  used by the http-c test suite, generated by
  `.devcontainer/run-tests.sh`).
- Load generator: `wrk` (2 threads, 64 connections,
  15s duration), invoked over loopback.
- Container: `http-c-build:ubuntu24` (Ubuntu 24.04 LTS, Linux 6.6.87.2-microsoft-standard-WSL2 x86_64).
- CPU: AMD Ryzen 7 PRO 7840U w/ Radeon 780M Graphics (16 cores visible to the container).

## Servers under test

| Server   | Source                                  | I/O model                          |
|----------|-----------------------------------------|------------------------------------|
| http-c   | this repository (commit `e6d1fd0`) | single-threaded epoll event loop, non-blocking OpenSSL, zero alloc per request |
| mongoose | github.com/cesanta/mongoose (609a0bd)    | single-threaded `poll` event loop, OpenSSL TLS adapter |

> lwIP is **not** included as a runtime peer. It is a TCP/IP stack
> targeted at MMU-less RTOS environments, not a Linux HTTP server: there
> is no native socket(2) backend on Linux, only an experimental TUN/TAP
> simulator (`contrib/ports/unix`) that provides a userspace IPv4 stack
> on top of a tap device. Comparing wire-line throughput against a
> kernel-bypass simulator is unfair in either direction. We instead note
> qualitative tradeoffs in the discussion below.

## Discussion

Both servers run on a single thread and use OpenSSL for the TLS layer, so
this measures *the framing + dispatch overhead* on top of a common TLS
record path rather than crypto throughput. The numbers above should be
read as relative, not absolute.

### http-c characteristics

- Static caller-supplied storage (no malloc/free on the request path).
- Bounded slot count (`HTTP_SERVER_HOST_CONNECTIONS`, default 256). When
  every slot is in use the listener is unregistered from epoll until a
  slot frees, providing strict back-pressure.
- Incremental request parser scans only the new bytes per chunk and
  short-circuits once `Content-Length` is known.
- Response is serialised in place into a per-slot send buffer; no
  intermediate copies.

### mongoose characteristics

- Linked list of connections; `mg_mgr_poll` walks the list each tick.
- Allocates request and response buffers per connection lifecycle.
- TLS adapter is well-tested and matches the OpenSSL setup.

### lwIP (qualitative note)

lwIP shines where http-c and mongoose cannot run at all: the lwIP HTTPD
on a 32 KB MMU-less microcontroller without an OS socket layer. Its
zero-copy PBUF chains and tight memory pools are tuned for that
environment. On Linux, where a real epoll/SO_REUSEPORT kernel stack is
available, lwIP's userspace TCP/IP stack adds latency rather than
removing it; that is why no production Linux server uses it.

## Reproducing

```sh
docker run --rm -v $(pwd):/work -w /work http-c-build:ubuntu24 \
    bash /work/benchmarks/run_bench.sh
```

Override the load shape via environment variables:

```sh
DURATION=30s THREADS=4 CONNECTIONS=256 \
    docker run --rm -e DURATION -e THREADS -e CONNECTIONS \
        -v $(pwd):/work -w /work http-c-build:ubuntu24 \
        bash /work/benchmarks/run_bench.sh
```
