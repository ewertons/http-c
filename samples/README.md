# http-c sample

A self-contained demo that starts the http-c HTTPS server and an http-c
HTTPS client **in the same process**, performs one `GET /` round trip
over TLS, and prints the HTTP traffic seen on both sides.

## What you get

```
http-c sample
  server cert: samples/certs/server.cert.pem
  server key : samples/certs/server.key.pem
  listening  : https://localhost:8084/

----- CLIENT sending request -----
GET / HTTP/1.1
Host: localhost:8084
User-Agent: http-c-sample/1.0
Accept: */*
Connection: close

----- SERVER received request -----
GET / HTTP/1.1
Host: localhost:8084
User-Agent: http-c-sample/1.0
Accept: */*
Connection: close

----- CLIENT received response -----
HTTP/1.1 200 OK
Content-Type: text/plain
Content-Length: 6

hello

Done.
```

The CR/LF line endings shown are exactly what gets serialised onto the
wire by `http_request_serialize_to` / `http_response_serialize_to`.

## 1. Generate a server cert and key

The sample needs a TLS server certificate and matching private key. A
helper script generates a self-signed pair (CN=`localhost`,
SAN=`DNS:localhost,IP:127.0.0.1`, valid 30 days) suitable for local
demos only:

```sh
./samples/scripts/generate_certs.sh
```

This writes:

- `samples/certs/server.cert.pem`
- `samples/certs/server.key.pem`

The script is idempotent: it will not overwrite existing files. To
regenerate, delete the two files first.

You can override the output directory:

```sh
./samples/scripts/generate_certs.sh /tmp/http-c-sample-certs
```

## 2. Build

From the repo root, configure once and build the `sample` target:

```sh
cmake -S . -B build
cmake --build build --target sample
```

The binary lands at `build/samples/sample`.

## 3. Run

The sample resolves cert paths from environment variables, falling back
to `samples/certs/*` **relative to the current working directory**. Run
it from the repo root:

```sh
./build/samples/sample
```

Or point it at a different cert/key pair:

```sh
HTTP_C_SERVER_CERT=/tmp/http-c-sample-certs/server.cert.pem \
HTTP_C_SERVER_KEY=/tmp/http-c-sample-certs/server.key.pem \
    ./build/samples/sample
```

The client trusts the server's cert directly (the same self-signed PEM
is loaded as the client's trust anchor), so no separate CA file is
needed.

## What the sample exercises

- `http_server_init` / `http_server_add_route` / `http_server_run_async`
  on the server side, with the host-class storage provider
  (`http_server_storage_get_for_server_host`).
- `http_endpoint_init` (client role) / `http_endpoint_connect` /
  `http_connection_send_request` / `http_connection_receive_response`
  on the client side.
- `task_run` to drive the client off the main thread, mirroring the
  pattern used by the test suite.
- A route handler that adds `Content-Type` and `Content-Length` headers
  to the framework-supplied default response.

## Troubleshooting

- **`http_server_init failed`** — the cert/key paths are wrong, or port
  `8084` is already taken. Re-run the cert script and verify the paths
  printed at startup.
- **`http_endpoint_connect failed`** with a TLS verify error — the cert
  file the client is trusting does not match the cert the server is
  presenting. Make sure both sides see the same `server.cert.pem`.
- **The cert expired** — the helper script issues 30-day certs. Delete
  `samples/certs/` and rerun the script.

## Security note

The cert generated by `generate_certs.sh` is **for local development
only**. It is self-signed, has no revocation story, ships its private
key in plaintext under the repo, and uses a hard-coded subject. Do not
use it for anything that touches a real network.
